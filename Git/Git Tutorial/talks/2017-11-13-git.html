Why using a Distributed Version Control System

  • Insure to always be able to have a working version of your project
  • Change your code without being worry about breaking something
  • Track your changes :
    • Go back to every step of development

    • Know who wrote what part of the code when
  • Team work
  • Use version even offline.

Why git

Pros

  • Robust: The linux kernel uses git !
  • Loosing data is difficult :

Cons

git is made to keep track of changes, and be able to revert them if needed, as we will see after.

Best Practice :

  • Split commit by Logical units of changes : \(x\) files modified to implement one behaviours

  • Don’t mix form and substance in one commit : reindentation separate from code

  • Don’t mix background modifications and feature : one commit to refactor you software

  • in oder to support a feature and one commit to implement the feature.

  • Make useful commit messages : explained what the patch does, not why you did it.

Revert a file with committed modification

I did something wrong ! I want to get my file back to a committed version !

git lola # Find the version
git diff <version> # Verify the content 
git checkout <version> <file/files> # don't forget the files !
git commit

Unless you know what you are doing, if you want to revert, always use git checkout with a version and filenames, if you do otherwise, it will not do what you intended !

Revert commits

You should check this link :

git revert <version> # Revert only this commit
git revert <version>..<version> # Revert this set of commits
git revert HEAD^ # Revert parent commit
git revert HEAD^^ # Revert grand parent commit
git revert HEAD^^..HEAD^ # Revert grand parent and parent commit

Understanding ^ and ~ : https://stackoverflow.com/a/23714994/1042370

Work with remote

The Fork model

Instead of all pushing on the same remote server, each user get a remote server and use merge(pull) request.

  • Fork the project on github
  • Make your modification as usual
  • Update your fork with the main repository
  • Push your modifications
  • Make a pull / merge request on github

Work with remote

Keep your fork up to date

git clone <address of your repo>
cd <project>
git remote add upstream <address of the original repo>
git fetch upstream
git pull upstream master
git push

Go Further

Best Practices

  • Commit soon and often
  • Never rewrite history (ex: git push --force) on a public repository : this is the only way to loose data
  • Branch a lot
  • Don’t commit big binary file : Every copy of the repository will have all versions of the file
  • Don’t commit generated files
